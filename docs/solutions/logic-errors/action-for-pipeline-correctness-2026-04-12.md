# `action_for` Silent Swallow and `AvailableDiscard` Order-Sensitive Equality

## Problem

Two correctness bugs formed a compound failure in the `action_for` → `available_actions` pipeline:

1. **`action_for` conflated two distinct outcomes.** When a strategy returned an action that was not in `state.available_actions`, `action_for` silently returned `None` instead of raising. This made a strategy bug look identical to an intentional abstention — callers could not distinguish them.

2. **`AvailableDiscard.__eq__` was order-sensitive.** Python's auto-generated `__eq__` for `@dataclass(frozen=True)` performs field-by-field comparison; for a `tuple[Card, ...]` field that means order matters. The discard candidates in `available_actions` are generated by `itertools.combinations()`, which produces a deterministic order. Any strategy that assembled a discard set in a different order would produce an `AvailableDiscard` that compared not-equal to every entry in `available_actions`, even though the same cards were present.

Together: a strategy returning a valid-but-differently-ordered discard would fall through the `not in available_actions` guard, and then `action_for` would silently return `None` rather than surfacing the mismatch.

## What Didn't Work

### Combined `if` condition

```python
# Before — in action_for:
if suggested_action is None or suggested_action not in available_actions:
    return None
```

Collapsed deliberate abstention and invalid actions into the same `None` return — callers could not distinguish them.

### Default dataclass `__eq__` on `AvailableDiscard`

```python
# Before:
@dataclass(frozen=True)
class AvailableDiscard:
    cards: tuple[Card, ...]
```

`@dataclass(frozen=True)` auto-generates `__eq__` as `self.cards == other.cards` — order-sensitive tuple comparison. A discard is a *set* of cards; order is irrelevant.

Note: the pre-existing test `test_adapter_checks_action` masked this bug by using `AvailableDiscard(cards=())` during `BIDDING`, where no discard actions exist regardless of order.

## Solution

### Fix 1 — `action_for` in `EngineAdapter`

Split the combined condition into two separate branches with different behavior:

```python
# packages/hundredandten-automation-engineadapter/src/hundredandten/automation/engineadapter/__init__.py

@staticmethod
def action_for(
    game: Game,
    identifier: str,
    decision_fn: Callable[[GameState], AvailableAction | None],
) -> AvailableAction | None:
    state = EngineAdapter.state_from_engine(game, identifier)
    suggested_action = decision_fn(state)
    if suggested_action is None:
        return None
    if suggested_action not in state.available_actions:
        raise StateError(
            f"decision_fn returned an action not in available_actions: {suggested_action}"
        )
    return suggested_action
```

Updated tests in `TestAdapterActionFor`:

```python
def test_adapter_checks_action(self):
    """Adapter raises StateError if the decision function returns an unavailable action"""
    game = arrange.game(EngineStatus.BIDDING, seed=SEED)
    active = game.active_round.active_player

    with self.assertRaises(StateError):
        EngineAdapter.action_for(
            game, active.identifier, lambda _: AvailableDiscard(cards=())
        )

def test_adapter_returns_none_when_decision_fn_returns_none(self):
    """Adapter returns None when the decision function returns None"""
    game = arrange.game(EngineStatus.BIDDING, seed=SEED)
    active = game.active_round.active_player

    action = EngineAdapter.action_for(game, active.identifier, lambda _: None)

    self.assertIsNone(action)
```

### Fix 2 — `AvailableDiscard` order-insensitive equality

Override `__eq__` and `__hash__` to use `frozenset` semantics:

```python
# packages/hundredandten-state/src/hundredandten/state/__init__.py

@dataclass(frozen=True, eq=False)
class AvailableDiscard:
    cards: tuple[Card, ...]

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, AvailableDiscard):
            return NotImplemented
        return frozenset(self.cards) == frozenset(other.cards)

    def __hash__(self) -> int:
        return hash(frozenset(self.cards))
```

`eq=False` suppresses dataclass auto-generation so the custom `__eq__` is not overwritten. `frozen=True` is retained for immutability; providing an explicit `__hash__` satisfies Python's requirement that objects with custom `__eq__` also define `__hash__`.

New test file `packages/hundredandten-state/tests/state/test_available_discard.py` covers:

- Same cards, same order → equal
- Same cards, different order → equal
- Different card sets → not equal
- Different lengths → not equal
- Non-`AvailableDiscard` comparison → not equal
- Empty discard → equal to another empty discard
- `hash()` consistent with `__eq__` for both same-order and different-order cases
- Instances with same cards collapse to one entry in a `set`

## Why This Works

`itertools.combinations()` generates discard tuples in a fixed iteration order. Any strategy assembling a discard from a different traversal produces a tuple in a different order. Using `frozenset` for equality makes membership in `available_actions` depend only on card identity, not order.

Raising `StateError` instead of returning `None` surfaces a strategy bug at the adapter boundary, where the contract is violated, rather than propagating it as a silent abstention.

## Prevention

1. **Separate semantically distinct `None` branches.** When a function can return `None` for multiple reasons, never collapse them into a single condition — callers need to handle each case explicitly.

2. **Audit `__eq__` on any wrapper type that holds a collection.** When a frozen dataclass wraps a `tuple` that represents an *unordered* domain concept (a set of cards, a bag of tokens), the auto-generated tuple `__eq__` will be semantically wrong. Override it with `eq=False` and implement `__eq__` and `__hash__` via `frozenset`.

3. **Test `in` membership, not just instantiation.** When writing tests for action wrapper types that hold collections, include an explicit test that verifies `AvailableDiscard((a, b)) in [AvailableDiscard((b, a))]` — this would have caught the order bug at introduction.

4. **When overriding `__eq__` on a frozen dataclass, always also define `__hash__`.** Python will set `__hash__ = None` if you define `__eq__` without `__hash__`, turning the dataclass into an unhashable type.
